## Acceptance Criteria

1. Summary, work-bullet, and wording actions have a real production click handler.
2. Each click dispatches exactly one typed targeted-regeneration command.
3. AI-enabled state permits execution; disabled state gives an explicit reason.
4. Running and validating states reflect the actual asynchronous runtime lifecycle.
5. The existing automation endpoint is invoked only after explicit click.
6. Command/request identity includes request ID, blocker IDs, target zones, CV version/content hash, effective brief hash, and selected evidence IDs.
7. Runtime response is normalized and validated before save.
8. Allowed mutation zones, preserved zones, evidence traceability, duplicate content, unsupported wording, and required fields are enforced.
9. Success saves a new CV version/hash and refreshes review, repair, workflow, and export state.
10. No-diff, blocked, stale, invalid response, runtime error, and disabled permission are visible and never returned as undefined.
11. Duplicate clicks are prevented.
12. A focused browser integration test exercises the real `App` and `ScreeningLab` path with a mocked local backend response.
13. No hidden AI call, automatic repair loop, review-rule change, export-rule change, prompt change, runtime-data change, persistence redesign, Phase 5 task, or additional READY task is introduced.
